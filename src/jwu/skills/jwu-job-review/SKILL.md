# jwu: ревью изменений ДО коммита/пуша

## Overview

**Фаза 4 в `jwu-start-job` — ВСЕГДА до коммита и пуша.** Идея: поймать проблемы пока
ничего не зафиксировано в истории, и не пушить «грязные» коммиты.

Скилл собирает локальный дифф (рабочее дерево + ещё не запушенные коммиты против
целевой ветки) + контекст задачи Jira и отдаёт всё это субагенту-ревьюверу.

## Аргумент и выбор субагента

Вызов: `/jwu-job-review <reviewer-subagent>` — где `<reviewer-subagent>` это имя
субагента-ревьювера (`subagent_type` для тула Agent).

Правила выбора:

- Если имя **передано аргументом** — используй его как есть, не подменяй.
- Если имя **не передано**:
  1. Посмотри на контекст (стек проекта, репозиторий PR, прошлые ревью в этой работе).
     Если очевидно подходит конкретный субагент-ревьювер — **предложи** его пользователю
     через AskUserQuestion как первый вариант.
  2. Если подходящего нет — предложи дефолтный **`reviewer-jwu-sample`** (он шипуется
     вместе с jwu и не привязан к конкретному стеку, ревьюит по универсальному чек-листу).
  3. **Всегда уточни у пользователя** имя ревьювера перед запуском — никаких автозапусков
     без явного подтверждения.
- В самом этом скилле **не хардкодь** список «правильных» ревьюеров под проекты —
  привязка проекта к ревьюверу должна жить в фазе 5 конкретного `jwu-start-job`-флоу
  или в пользовательских настройках, не здесь.

## Шаги

1. **Найди работу.**
   - Подхвати текущую работу (если контекста нет — через **jwu-resume-job**): нужен
     `JOB_ID` и ключ задачи.
   - PR на этой стадии **может ещё не существовать** — это нормально, ревью делается
     ДО коммита/пуша. Если PR уже привязан к работе — забери его метаданные (см. шаг 4)
     для контекста, но дифф всё равно бери локальный.

2. **Подтверди субагента-ревьювера** по правилам из «Аргумент и выбор субагента» выше.

3. **Собери контекст задачи.**
   - `jwu task <KEY> --json` — вытащи `summary`/`description`/`status`/`comments`/
     `links`/`branches`/`pull_requests`. Не дампь сырой JSON в промпт: оставь
     заголовок, описание и значимые комменты (требования/договорённости/решения).

4. **Собери контекст PR (если привязан) — опционально.**
   - Если PR уже создан: `jwu pr <PR_ID> --project <KEY> --repo <slug> --json` —
     `title`, `description`, `source_branch`, `target_branch`, `state`,
     `conflicted`/`can_merge`, и **`comments`** (комменты ревью с `file:line`),
     чтобы ревьювер видел уже сделанные внешние замечания и не дублировал их.
   - Если PR нет — пропусти этот шаг, target_branch определи отдельно (см. шаг 5).

5. **Собери ЛОКАЛЬНЫЙ дифф изменений vs целевой ветки.**

   **Цель:** ревью идёт ДО коммита и ДО пуша. Источник правды — локальное рабочее
   дерево + ещё не запушенные коммиты текущей ветки, **не** уже запушенный PR.

   1. Определи **целевую ветку** (`<TARGET>`):
      - из метаданных PR (`target_branch`), если PR есть;
      - иначе из ROADMAP/конвенции проекта (обычно `develop`/`main`/`release-X.Y`);
      - если непонятно — **спроси пользователя**, не угадывай.
   2. Обнови ссылку на ремоут: `git fetch origin <TARGET>` (на ошибке — сообщи, не вались).
   3. Возьми **всё, что отличает рабочее дерево от `origin/<TARGET>`**:
      ```bash
      git diff origin/<TARGET>                # дерево + индекс vs target (committed+staged+unstaged)
      git status --short                       # untracked отдельно (строки `??`)
      ```
      В промпт ревьюверу клади обе части. Untracked-файлы тоже важны — новые модули,
      миграции, конфиги легко проворонить, если смотреть только `git diff`.
   4. Если изменений вообще нет (`git diff origin/<TARGET>` пустой и untracked нет) —
      сообщи пользователю «ревьюить нечего» и **не запускай** субагента.
   5. Большой дифф можно резать по файлам/чанкам, но в промпт должен пойти **весь**
      дифф (включая untracked), чтобы покрыть пункты «миграции», «потеря правок из
      target», «нейминг».

6. **Запусти субагента-ревьювера** через тул `Agent` с `subagent_type = <тот, что подтверждён>`.
   В `prompt` положи структурированный user-prompt такого вида:

   ```
   Проверь локальные изменения на соответствие задаче из Jira. Это ревью ДО коммита/пуша.

   ## Задача из Jira: <TICKET_KEY>
   Заголовок: <summary>
   Статус: <status>

   Описание:
   <description>

   Ключевые комментарии:
   <выжимка по делу: требования / решения / договорённости>

   ## Контекст изменений
   Целевая ветка: <TARGET>
   Текущая ветка: <CURRENT>
   PR: #<PR_ID> (<source>→<target>), state=<state>, can_merge=<can_merge>, conflicted=<conflicted>
   <если PR ещё нет — напиши «PR пока не создан»>

   Уже сделанные внешние замечания на PR:
   <если есть — список threads с file:line; если PR нет — «нет»>

   ## Локальный diff (рабочее дерево vs origin/<TARGET>)
   ```diff
   <вывод git diff origin/<TARGET>>
   ```

   ## Новые untracked-файлы
   <git status --short, только строки `??`; если пусто — «нет»>
   ```

   Обязательно подставь ключ задачи в «## Задача из Jira» — это критично для пункта
   «соответствие задаче».

7. **Залогируй ревью в работу — это ОБЯЗАТЕЛЬНЫЙ шаг, не пропускать.**
   Все записи делаются через `jwu job add <JOB_ID> --kind <тип> "<текст>"` (см. также
   **jwu-track-job**).

   ### КРИТИЧНО: записи неизменяемы
   `jwu` НЕ поддерживает редактирование записей работы — `jwu job add` это append-only.
   Поэтому **шапка ревью (`--kind review`) должна быть самодостаточной**: прочитав её одну
   через сутки/неделю, человек должен понимать, что нашёл ревьювер, без чтения деталей.

   ### Якоря и итерации
   - **Итерация цикла** — это порядковый номер прогона ревью в рамках текущей работы (job).
     Перед записью посчитай существующие `--kind review` в работе: `jwu job show <JOB_ID> --json`
     → отфильтровать `kind == "review"`; новая итерация = `len(...) + 1`.
   - **Якоря замечаний** — короткие коды, привязывающие детальные записи к шапке:
     - `[B<n>]` — блокер (Blocker), номеруется с 1 по порядку важности.
     - `[I<n>]` — важная доработка (Important).
     - `[M<n>]` — мелкое замечание (Minor).
     Якоря **сквозные внутри одного прогона ревью** (одной `review`-записи) и **обязательно**
     повторяются в начале текста каждой связанной `remark`/`bug`/`warning`/`constraint`-записи —
     так фильтрация по типу даёт детали, привязанные к шапке.

   ### Минимальный набор записей по результату прогона

   1. **Шапка ревью** — `--kind review`. Ровно одна запись на прогон. Формат:

      ```
      Ревьюер: <reviewer-subagent>; ветка <CURRENT>→<TARGET>; PR #<PR_ID|—>; итерация <n>;
      вердикт: <можно коммитить|нужны доработки|есть блокеры>;
      блокеры: N, важные: M, мелкие: K.
      Замечания:
      [B1] <короткий заголовок блокера>
      [B2] <…>
      [I1] <короткий заголовок важной доработки>
      [I2] <…>
      [M1] <короткий заголовок мелкого замечания>
      …
      ```

      Если PR ещё не создан — поле `PR #<PR_ID>` ставим как `PR #—`. Вердикт «можно
      коммитить» означает, что пользователь может переходить к **Фазе 5 (коммит/пуш)**
      по `jwu-start-job`.

      Заголовок одного замечания — **одна строка, до ~80 символов**, по сути проблемы
      («N+1 в `OrderService.list_for_user`», «нет проверки прав в `views/admin/users.py:42`»).
      Если замечаний нет — блок «Замечания:» опусти и в счётчиках поставь нули.

   2. **Каждый блокер** — `--kind constraint` (⛔). Текст начинается с якоря шапки и содержит
      дословную формулировку ревьювера + место (файл/строка) + почему это блокер:
      `[B1] <дословная цитата>. Где: <file:line>. Почему блокер: <…>.`

   3. **Каждый баг, найденный ревьювером** — `--kind bug`. Формат:
      `[B<n>|I<n>] <дословный текст замечания>. Где: <file:line>. Симптом: <…>.`
      Якорь выбирается по серьёзности (баг-блокер → `[B*]`, баг-важное → `[I*]`).

   4. **Каждая важная/мелкая доработка** — `--kind remark`. Формат:
      `[I<n>|M<n>] <дословный текст замечания>. Где: <file:line>.`

   5. **Потенциальные риски / «осторожно с X»** — `--kind warning`. Якорь — `[I<n>]` или `[M<n>]`
      в зависимости от серьёзности.

   6. **Закрой фазу 4** — `--kind phase "Фаза 4 — ревью изменений (итерация <n>)" --status done`
      (одна запись на прогон). Только после успешного прогона можно идти в **Фазу 5
      (коммит/пуш)** из `jwu-start-job`.

   ### Пустой прогон
   Если ревьювер вернул «можно мержить» и замечаний ноль — всё равно пишутся **две** записи:
   `review` (с нулевыми счётчиками и без блока «Замечания:») и `phase done`. Это валидный
   результат — он должен быть зафиксирован.

   ### Чего НЕЛЬЗЯ
   - Не дампь весь ответ ревьювера в одну запись.
   - Не пиши шапку без блока «Замечания:» если найденные пункты есть — она должна быть
     самодостаточной (редактировать её потом нельзя).
   - Не вешай детальные записи без якоря — связь шапки и деталей потеряется.
   - Не переиспользуй якоря из прошлой итерации — каждая `review`-запись имеет свой набор
     `[B1]…/[I1]…/[M1]…`, локальный для своей итерации.

8. **Покажи пользователю краткое резюме** (вердикт + блокеры + важные доработки).
   Дальше:
   - есть блокеры/важные → **возврат в Фазу 3** (новая итерация правок), потом снова Фаза 4;
   - вердикт «можно коммитить» → **переход в Фазу 5** (коммит и пуш) по `jwu-start-job`.
   Пушить «без ревью» — запрещено.

## Что НЕ делать

- **НЕ запускай ревью после коммита/пуша как формальность.** Канонический порядок:
  правки → ревью (этот скилл) → коммит. Если уже закоммитил — ОК, но ревью всё равно
  обязательно ДО пуша.
- Не хардкодь имена проектно-специфичных ревьюеров в этом скилле — они приходят
  аргументом или выбираются пользователем в момент запуска. Единственное явное
  упоминание здесь — дефолтный `reviewer-jwu-sample`, который шипуется вместе с jwu.
- Не запускай ревьювер «по памяти» без подтверждённого имени.
- Не подменяй контекст задачи своим пересказом — бери из `jwu task`/`jwu pr`.
- Не подменяй локальный дифф диффом из Bitbucket — там может быть устаревшее состояние,
  а правки в рабочем дереве вообще не попадут в ревью.
- Не редактируй код по итогам ревью без явного подтверждения пользователя
  (правило проекта — правки только после ОК).

## Связанные скиллы и агенты

- **jwu-start-job** — заводит работу и фиксирует план по фазам. По канону: Ф4 — ревью
  (этот скилл, ДО коммита), Ф5 — коммит/пуш.
- **jwu-resume-job** — подхватывает работу после потери контекста перед запуском ревью.
- **jwu-track-job** — фиксирует фазу ревью и его замечания в работе.
- **jwu-commit-message** — текст коммита по локальным изменениям; вызывается в Фазе 5
  ПОСЛЕ успешного ревью.
- **reviewer-jwu-sample** — дефолтный универсальный субагент-ревьювер из поставки jwu;
  используется, когда никакого более специфичного ревьювера для этого проекта нет.
