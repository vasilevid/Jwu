# jwu: старт и трекинг работы (job)

## Overview

Каждый **цикл работы** над задачей трекается отдельным объектом **работа (job)** в `jwu`: прогресс
по фазам, записи и замечания, связь с задачей (1) и PR (N). Новый раунд правок (напр. после ревью)
= новая работа. Пока просят «проанализировать» — **только анализ и план, никаких правок кода**.
Аргумент — ключ задачи (напр. `PROJ-399`); голый номер без префикса — спроси полный ключ.

## Шаги

1. **Карточка задачи целиком:** `jwu task <KEY>` (поля: `status`, `description`, `comments`, `links`,
   `pull_requests`, `branches`, `commits`, `notes`, плюс секция «Работы»; `--json` если нужна структура).
   **Сначала смотри `status`.** Done / Closed / Resolved → не планируй правки, сообщи и спроси.

2. **Создай работу (всегда новую):**
   - `jwu jobs --task <KEY>` — **обязательно** проверь существующие работы. Если они есть —
     **предупреди пользователя** (перечисли id/заголовки/статусы), но **всегда начинай новую**:
     `jwu job start <KEY> --title "<суть цикла>"`. Существующие работы НЕ продолжай.
   - Запиши id новой работы — он нужен для всех `jwu job add/link`.

3. **Контекст — экономно, без фанатизма** (не дампь весь JSON, бери суть):
   - **Комменты** — вычитай все, выдели требования/договорённости/решения. **Сверяй время коммента
     с последними правками ветки:** коммент ПОСЛЕ последнего пуша — актуален и в ветке ещё НЕ учтён;
     предложенные ревьюером варианты надо попробовать, а не отмахнуться.
   - **PR** — из `pull_requests`/секции «Работы»; детали/конфликт нужного: `jwu pr <id> --project <KEY> --repo <slug>`. **Замечания ревью берутся отсюда** — в выводе (и в `--json` поле `comments`) идут все комментарии PR: верхний уровень и ответы (`depth`), inline с `file:line`, свежие треды первыми. Если `NEEDS_WORK`, а `comments` пуст — проверь `--project/--repo` (без них берутся дефолтные из конфига). Привяжи к работе: `jwu job link <JOB_ID> --pr <id> --project <KEY> --repo <slug>`.
   - **Связанные задачи** — открывай `jwu task <KEY>` по `links` только если на них **явно** ссылаются. Не уходи рекурсивно вглубь.
   - **Вложения (скриншоты багов, логи, файлы, документы)** — список и счётчики по видам даёт `jwu attachments <KEY>` (он же в `jwu task` секцией «Вложения» и в поле `attachments` у `--json`; видео — только в списке, не качаются).
     **Если вложения есть — спроси пользователя ПЕРЕД анализом** (вопросом вида «Нашёл N изображений, M логов и K документов — проанализировать их?») и **дождись ответа**. Не качай молча.
     После согласия скачай: `jwu attachments <KEY> --download` (по умолчанию image/log/doc/archive в `<tmp>/jwu/<KEY>/`; сузить — `--kind image --kind log`). Команда печатает локальные пути — открывай их через **Read** (изображения читаются визуально, логи/JSON/pdf — как текст; архивы распаковываются рядом в `*.extracted/`). Связывай увиденное (текст бага на скрине, стек в логе) с требованиями из комментов.

4. **Определи:** что сделать и **в каком проекте/репо** (по dev-панели / `branches` / `pull_requests` / префиксу ключа).

5. **План по КАНОНИЧЕСКИМ ФАЗАМ** (фазы процесса, не «список задач»; пункты раскладываются ВНУТРИ фаз).
   Порядок фиксированный, фиксируй фазы записями работы `jwu job add <JOB_ID> --kind phase`:
   - **Фаза 1 — мердж/устранение конфликта. ВСЕГДА первой.** PR/ветка в конфликте → сначала смердить целевую ветку и разрулить (`can_merge`). Остальное — после.
   - **Фаза 2 — анализ текущей реализации + план правок** (учесть и **попробовать** актуальные предложения ревьюеров).
   - **Фаза 3 — правки и фиксы** по согласованному плану.
   - **Фаза 4 — ревью изменений. ВСЕГДА ДО коммита/пуша.** Запускается через скилл **jwu-job-review** субагентом-ревьювером **по локальным изменениям** (рабочее дерево + ещё не запушенные коммиты vs. целевая ветка), чтобы поймать проблемы пока ничего не зафиксировано в истории. Имя субагента **не хардкодится** — его передаёт пользователь аргументом скилла (`/jwu-job-review <reviewer-subagent>`). Если из контекста (стек, репо, прошлые ревью) очевидно подходит конкретный субагент — **предложи** его как кандидата, но **всегда уточни у пользователя** и запускай только после подтверждения. Если кандидата нет — используется дефолтный `reviewer-jwu-sample`. Замечания фиксируй в работу через **jwu-track-job** (шапка `--kind review`, детали `remark`/`bug`/`warning`/`constraint`). Если ревьювер вернул блокеры или важные доработки — **возвращайся в фазу 3** (новая итерация правок), потом снова Фаза 4. На Фазу 5 переходим, **только когда ревьювер сказал «можно мержить»** или остались только мелкие замечания, которые пользователь явно решил оставить.
   - **Фаза 5 — коммит и пуш/обновление PR.** **ВСЕГДА последней.** Делается только после успешного прохождения Фазы 4. Коммит/пуш — по правилу проекта, после явного подтверждения пользователя.
   Большую задачу веди постепенно: начни с фазы 1, дальше по одной.
   Замечания ревьюеров фиксируй как `jwu job add <JOB_ID> --kind remark "<...>"`; явные запреты («не трогать X», «гонять без Y», «не пушить в Z») — `--kind constraint` (⛔ ЗАПРЕТ).

   **Канонический порядок ВСЕГДА: правки → ревью → коммит.** Не наоборот. Если коммит уже случился без ревью — всё равно прогоняй Фазу 4 до пуша; пушить «грязный» коммит без ревью **запрещено**.

   **Как вести записи по ходу работы** — это отдельная дисциплина: типы `phase` (со статусами), `bug` / `bug-resolved` (история «нашли → исправили»), `test-pass` / `test-fail` (прогоны тестов), `warning`, плюс фразовые триггеры. Полная таксономия и правила — в скилле **jwu-track-job**; обращайся к нему всякий раз, когда нужно что-то записать в работу во время исполнения.

6. **Покажи план пользователю.** По завершении цикла — `jwu job done <JOB_ID>`.

## КРИТИЧНО — подтверждение правок

- Пока просят анализ — **не редактируй код, не коммить, не пушь**.
- Любые правки/изменения — **только после явного подтверждения** пользователя (план/диф).
  Правило проекта (нельзя коммитить и пушить без подтверждения) распространяется и на правки.

## Заметки

- Read-only по Jira/Bitbucket: только чтение + запись работы в локальную память (`jwu job …`).
- Прогресс трекай через `jwu job` (фазы/записи/связи с PR); разовую заметку можно `jwu note <KEY>`.
- Статусы важнее устаревших комментов — всегда сверяйся с текущим `status`.
- Завершённые/текущие работы видны при открытии задачи (`jwu task`) и PR (`jwu pr`), и в дашборде (вкладка «Работы»).
- Экономь контекст: нужное из карточки, без обхода всех связей и всех PR.
