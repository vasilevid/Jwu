# jwu: 4test — инструкция для тестировщиков по тому, что сделали

## Когда

После того как правки готовы (обычно перед/после коммита) и нужно **отдать задачу в тест**:
сформировать чёткую, пошаговую инструкцию проверки для QA. Скилл **показывает текст** для
копирования в Jira-комментарий — сам в Jira ничего не постит, git/jwu не меняет (по
подтверждению может сохранить копию в работу, см. шаг 9).

Это **абстрактный** скилл: суть фикса и проверяемое поведение он выводит сам из работы,
задачи и диффа, а стенд-специфику и важные нюансы **пользователь передаёт текстом** в
аргументах вызова.

## Главный принцип

Тестировщик **не читает код**. 4test — это перевод «что мы сделали в коде» в
«какие действия выполнить и что при этом должно произойти». Поэтому:

- **Явная последовательность действий.** Всё нумеровано, по шагам, в порядке выполнения.
  Без «проверить, что всё работает» — только конкретные шаги.
- **Без технических аспектов.** В шагах НЕ упоминать классы, функции, поля кода, имена
  миграций, таблицы БД, эндпоинты, названия методов. Только наблюдаемое поведение.
  (Имя ветки и URL стенда — не «реализация», их оставляем: они нужны для подготовки.)
- **UI-проверяемое → описывать в терминах интерфейса.** Где нажать, что откроется, что
  увидеть. Назвать конкретную поверхность: РМО оператора, мобильное приложение, виджет
  посетителя, админка. Пример: «открыть на чате попап „Категории“ → категория отмечена
  галочкой».
- **Не видно в UI → логи.** Если результат нельзя увидеть в интерфейсе, дать **конкретную
  строку лога**, которую искать, и где смотреть (какой сервис/лог). Строки берутся из
  диффа (тексты `logger.*`), но подаются как «что искать», без объяснения реализации.
  Пример: «в логах при успехе — `Chat was sent to external handler`».
- **Каждый проверочный шаг — с ожидаемым результатом** (`ОЖИДАНИЕ: …`).

## Источники (что собрать перед составлением)

1. **Что сделали — работа (job):**
   - `jwu jobs --task <KEY> --json` → активную/последнюю → `jwu job show <JOB_ID> --json`.
   - Извлечь: суть фикса (`bug` → `bug-resolved`, `decision`, `phase`), и — критично —
     **ловушки проверки** (`warning`, `constraint`, `decision`): то, что влияет на то, КАК
     проверять, иначе тест даст ложный результат.
2. **Что это для пользователя — задача:** `jwu task <KEY>` (описание, комментарии,
   acceptance). Отсюда — наблюдаемое поведение и формулировка «что тестируем».
3. **Что изменилось на поверхности — дифф:** `git diff <target>...HEAD` (непушенные
   правки). Нужен для двух вещей: какие **UI-поверхности** затронуты и какие **строки логов**
   добавлены/изменены (искать `logger.`, `log.`, `LOG.*`, текст в кавычках рядом). Если
   таргет/ветка неясны — `git diff HEAD`, `git log --oneline -15`.
4. **Стенд-специфика и нюансы — свободный текст от пользователя** (аргументы вызова):
   URL стенда и доступы, имя ветки (если не из git), какие настройки аккаунта включить,
   устройства/каналы, важные моменты из задачи, известные ловушки. **Это приоритетный
   источник стенд-данных** — не выдумывай их сам.

## Структура сообщения (канон, выведен из реальных 4test)

```
4TEST — <KEY>: <кратко, что проверяем> (для тестировщиков)

ЧТО ТЕСТИРУЕМ
<1–3 предложения простым языком: какое наблюдаемое поведение/эффект проверяем.
 Без техники — перевод «что сделали» в «что должно работать для пользователя».>

ВАЖНО ПЕРЕД СТАРТОМ        ← только если есть ловушки/false-positive
<условия, при которых тест даст ЛОЖНЫЙ результат. Напр.: «тот, кто сам выставил X,
 видит его локально, а не из дельты — проверять только на втором операторе».>

ПОДГОТОВКА
1. Залить ветку <branch> на стенд <стенд>.
2. Включить/настроить <нужные опции аккаунта>.
3. Завести <аккаунты/устройства/каналы> и создать предусловие (<чат/диалог/данные>).

ОСНОВНОЙ СЦЕНАРИЙ
4. <действие в UI>. ОЖИДАНИЕ: <что видно/в логах>.
5. <действие>. ОЖИДАНИЕ: <…>.
...

КОНТРОЛЬ РЕГРЕССА БАГА     ← для багфиксов: доказать, что фикс работает
N. На сборке БЕЗ ветки шаги <…> воспроизводят баг; с веткой — нет.

РЕГРЕСС СОСЕДНЕЙ ФУНКЦИОНАЛЬНОСТИ
M. <смежные сценарии работают как раньше; лишних эффектов нет>.
```

Не все блоки обязательны: `ВАЖНО ПЕРЕД СТАРТОМ` — только при наличии ловушек;
`КОНТРОЛЬ РЕГРЕССА БАГА` — для багфиксов (не для новой фичи); количество шагов — по сути.

## Алгоритм

1. **Определить задачу `<KEY>`** — из имени ветки регуляркой `^[A-Z][A-Z0-9]+-[0-9]+`,
   либо из активной работы (`jwu jobs --status active --json`), либо спросить. Не выдумывать.
2. **Собрать источники** (раздел выше): работа → задача → дифф → пользовательский текст.
3. **Сформулировать «ЧТО ТЕСТИРУЕМ»** — наблюдаемый эффект для пользователя, простым языком.
4. **Выписать ловушки → «ВАЖНО ПЕРЕД СТАРТОМ»** из `warning`/`constraint`/`decision` работы
   и из пользовательского ввода (что сделает тест ложным).
5. **ПОДГОТОВКА** — нумерованные шаги: ветка на стенд, настройки аккаунта, аккаунты/устройства,
   предусловие. Стенд/ветку/настройки брать из пользовательского текста и git/работы.
6. **ОСНОВНОЙ СЦЕНАРИЙ** — для каждого проверяемого поведения шаг + `ОЖИДАНИЕ`. Решить по
   каждому: **видно в UI?** → описать UI-шаг и что увидеть. **Не видно?** → дать конкретную
   строку лога (из диффа) и где её искать.
7. **Багфикс?** → добавить `КОНТРОЛЬ РЕГРЕССА БАГА`: на сборке без ветки баг воспроизводится
   (шаги), с веткой — пропадает. Это превращает «починили» в проверяемый критерий.
8. **Добавить `РЕГРЕСС СОСЕДНЕЙ ФУНКЦИОНАЛЬНОСТИ`** — что рядом не должно сломаться.
9. **Вывести готовый текст в код-блоке** для копирования в Jira. Затем **предложить**
   сохранить копию в работу: `jwu job add <JOB_ID> --kind note "<текст 4test>"` — по
   подтверждению (как в реальных примерах хранится `note`). В Jira **не постить**, git не трогать.

## Чего не хватает — спрашивай, не выдумывай

Если для подготовки нет данных (URL стенда, имя ветки, какие настройки включить, доступы) —
**не сочиняй**. Либо спроси пользователя, либо вставь явный плейсхолдер `<…>` и предупреди,
что его надо заполнить. Лучше дырка в шаблоне, чем выдуманный стенд/настройка.

## Частые ошибки

| Ошибка | Правильно |
|---|---|
| Технические детали в шагах (классы, поля, миграции, эндпоинты) | Только наблюдаемое поведение и действия в UI |
| «Проверить, что всё работает» одним пунктом | Явные нумерованные шаги, каждый с `ОЖИДАНИЕ:` |
| Проверка не видна в UI, а лог не указан | Дать конкретную строку лога из диффа и где её искать |
| Нет «ВАЖНО ПЕРЕД СТАРТОМ», хотя есть ловушка | Вынести условие ложного результата явно (напр. «только на втором операторе») |
| Багфикс без контроля регресса | Показать: без ветки баг есть, с веткой нет |
| Выдуманный стенд/ветка/настройки | Из пользовательского текста и git; нет — плейсхолдер + предупреждение |
| Шаги «по памяти разговора» | Только из работы/задачи/диффа/пользовательского ввода |
| Сам постим в Jira / делаем git | Только печатаем текст; постит/коммитит пользователь |

## Связанные скиллы

- **jwu-resume-job** — если контекст работы потерян, сначала подхвати лог (`jwu job show`),
  потом составляй 4test (суть фикса и ловушки берутся из записей работы).
- **jwu-track-job** — сохранить готовый 4test в работу записью `--kind note`.
- **jwu-commit-message** — соседний «генератор текста для копирования»: тоже только печатает,
  ничего не коммитит/не постит.
